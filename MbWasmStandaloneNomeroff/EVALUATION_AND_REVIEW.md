# Оценка системы, предложения по улучшению и Code Review

## 1. Оценка: Raspberry Pi 3

**Вердикт: Raspberry Pi 3, скорее всего, будет недостаточен для полноценной работы системы.**

### Аргументы

| Фактор | Raspberry Pi 3 | Требования системы |
|--------|----------------|--------------------|
| **CPU** | ARM Cortex-A53 1.2 GHz × 4 | nomeroff-net использует нейросети (YOLOv8, RNN/OCR), PyTorch |
| **RAM** | 1 GB | PyTorch + OpenCV + модели — несколько ГБ |
| **GPU** | Нет выделенного GPU | Нейросети без CUDA на CPU в 10–50× медленнее |
| **Опыт** | Описаны решения на Pi 4 (с оптимизацией) | Pi 3 слабее Pi 4 примерно в 2 раза |

### Возможные варианты

1. **Raspberry Pi 4 (4–8 ГБ RAM)** — минимально разумная платформа для работы с nomeroff-net на CPU.
2. **Jetson Nano / Jetson Orin** — если нужна работа в реальном времени с GPU.
3. **Ноутбук x64 с GPU** — текущий вариант для «на ходу» максимально оптимален.

### Если всё же пробовать на Pi 3

- Уменьшить разрешение кадров (например, 640×480).
- Понизить FPS (1 кадр в 5–10 сек).
- Использовать упрощённый pipeline nomeroff-net (если есть опция).
- Рассмотреть замену на Tesseract/лёгкий OCR без нейросетей (с меньшей точностью).

## 2. Предложения по улучшению

### Функциональность

1. **Экспорт отчётов** — выгрузка записей в CSV/Excel за период.
2. **Фильтр и поиск по номерам** — в модуле просмотра БД.
3. **Настройка чувствительности распознавания** — порог уверенности, чтобы снизить ложные срабатывания.
4. **История тревог** — отдельная таблица/страница по совпадениям с watchlist.
5. **Офлайн-режим** — очередь записей при отсутствии связи с API для последующей отправки.

### Технические улучшения

1. **Кэширование настроек** — уменьшить частые вызовы `localStorage` через JS interop.
2. **Retry при сбое API** — повтор попытки при временных сетевых ошибках.
3. **Health check** — периодическая проверка доступности Python API, GPS, Interbase.
4. **Логирование в файл** — для отладки в полевых условиях.
5. **Резервное копирование БД** — автоматический бэкап Interbase по расписанию.

### UX

1. **Индикатор качества GPS** — количество спутников, точность.
2. **Подсветка/звук при слабом GPS** — предупреждение, что координаты могут быть неточны.
3. **Ручная корректировка координат** — при ошибке GPS для уже сохранённой записи.

## 3. Code Review

### Сделанные исправления (в этой сессии)

| Проблема | Решение |
|----------|---------|
| GPS в рабочем режиме — браузер вместо реального устройства | `MaybeSaveRecords` использует только `Records.GetGpsPositionAsync()` (Nomeroff.Gps.Api) |
| Фото не попадает в БД в рабочем режиме | `transaction.Commit()` в `NomeroffInterbaseService` вызывается всегда; вставка FOTO вынесена из блока с проверкой GPS |
| «Автостарт» и «Не записывать без GPS» не сохранялись | MudSwitch с `CheckedChanged` — сохранение в localStorage сразу при переключении |
| Вставка FOTO только при наличии GPS | FOTO и GEO вставки разделены; Commit выполняется независимо от наличия GPS |

### Рекомендации по коду

1. **RecordsService.SaveRecordAsync**
   - Добавить логирование при `!response.IsSuccessStatusCode` и при исключениях.
   - Рассмотреть повтор запроса при временных ошибках.

2. **SettingsService**
   - Кэшировать `AutoStart` и `SkipSaveWithoutGps` (как `_cachedApiBaseUrl`) и сбрасывать кэш при `Set*`.

3. **Home.razor**
   - Класс `GpsPos` можно удалить — больше не используется в `MaybeSaveRecords`.

4. **File.razor**
   - `_saveVideoToDb` — загружать из `SettingsService.GetSaveVideoFramesToDbAsync()` при инициализации и сохранять при изменении.

5. **NomeroffInterbaseService**
   - Убрать `Console.WriteLine` в production, оставить только `_logger`.

6. **Общее**
   - Добавить валидацию URL API в настройках (формат, доступность).
   - Унифицировать обработку ошибок: централизованный сервис/компонент для отображения ошибок.
